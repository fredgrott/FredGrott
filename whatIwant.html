Okay a little agile and computer history background. Okay the moeny computation cost amount per hour use to be high for computer machinery and low for human programmers way back in the 1940s and 1950s. the other trend was that the User Interface code as a percentage was smaller than the percentage of code pertaining to the business logic classes.

At places like IBM,etc they came up with development process of waterfall. You wrote a specification-in-stone not-to-be-changed and than would habve this long drawn out process of development because the human hours were lower in cost per hour than the computing hour costs. There was no incentive to change as in enterprise the user are not the dictators of change and the cost structure was such that saving human developer hours would not statistically impact totoal project costs at all.

That all changed when we got faster CPUs and faster data storage units in that now the costs per hour for code development is higher than the operation costs per hour of that computing device. Than they came up with things like TDD to get into having a process that would try and discover the code -product variations that would be most likely in the process of eliminating them to produce a steady state fot he system composed opf the devloping application and the devlopers whereas the development goes at a fast clip to a successful finish.

The problem is unit testing is not the contextof the code-base its SEO-like in that it only indicates keyword-like value of the individual coding units. It will not have any contextual value implied by any unit test data unless the developer separates the code-base into business logic classes and UI classes, and only than if the developer only applies TDD unit testing to the business logic classes.

This is the underlying problem in all Agile process in that they ignore the underlying different contexts of the different areas of a typical application. The other trend involved in pointing to a miss-application of agile has to do with the imnpact of applications being used by millsions and billions as opposed to a few hundred and thousands in the 1940s and 1950s. Now, ew haev the quality of the UI as a factor in gaining market share not to mention all the UI potimization we do on mobile to deal with memory and cpu limiation constraints.

Compare Mobile to Enterprise and Web applications much you would compare a great Poem, PLay, and Novel. You specifcally combine gammar, syntax, and context in specfic implementations ina poem that are differnt than in a Play or a Novel. This is the same for Mobile in that it is vastly different application of code development grammar, syntax, and context than enterpise or applications devloped int he distnat past of software development. This is why youor CTO will not find data to counter my thesis because the trends and their basis in software development and computer development history is against the CTO's own counter thesis.
